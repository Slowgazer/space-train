# 游戏策划辅助 Skill

## 核心原则

在实现任何游戏功能前，必须先搜索多种实现方式，列出给用户做决定，然后才能开始编码。

## 流程

```
用户提出需求
    ↓
搜索 2-4 种实现方式（库、框架、原生方案等）
    ↓
每种方式给出：
  - 简要原理说明
  - 适用场景
  - 优缺点
  - 代码量级估计
  - 性能影响
    ↓
让用户选择（或提出修改方向）
    ↓
用户确认后，开始实现
```

## 触发条件

任何时候用户提出以下类型的需求，必须执行此流程：

- 添加新游戏功能（射击、移动、碰撞等）
- 实现视觉效果（粒子、动画、着色器）
- 选择技术方案（用什么库、什么架构）
- 设计游戏系统（背包、升级、对话、存档）
- 优化性能

## 输出格式示例

```
## 实现方式对比

### 方式 A：[方案名称]
- **原理**：[一句话说明核心实现逻辑]
- **适用**：xx 场景
- **优点**：[2-3 点]
- **缺点**：[1-2 点]
- **代码量**：~xxx 行

### 方式 B：[方案名称]
...

你想用哪种方式？或者你有其他想法？
```

## 注意事项

- 不要默认选择"最简单"的方式，要让用户自己决定
- 如果某种方式明显不适合，先说明原因再排除
- 实现后如果遇到问题，先回顾当初选择的原因再调整
- 用户明确说"就用 XX 方式"后才开始写代码
